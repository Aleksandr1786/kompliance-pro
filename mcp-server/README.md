# MCP-сервер КомплаенсПро — v0.3 (read + write, этап 2)

## Статус

4 инструмента. Три read-only (как в v0.2). Четвёртый — `generate_document_package`
— умеет запускать штатную генерацию документов, включая работу при
**закрытом** основном приложении. Защищён тремя независимыми предохранителями
(флаг, бэкап, блокировка) — см. ниже.

## Файлы и куда они кладутся

```
kompliance/                    ← корень проекта
├── main.js                    ← использует doc-generation.js через withLock
├── lock-utils.js              ← файловая блокировка (CommonJS)
├── doc-generation.js          ← вынесенная логика генерации (CommonJS)
├── mcp-write-bridge.js        ← мост для MCP: открывает lowdb, бэкап, вызывает generateDocsForClient
├── generator.js
├── gen_p1.js, gen_p2.js, ...
└── mcp-server/
    ├── server.js              ← сам MCP-сервер (ESM)
    ├── package.json
    └── readiness-calc.js      ← КОПИЯ src/modules/readiness-calc.js
```

`mcp-write-bridge.js` — CommonJS и живёт **в корне**, не в `mcp-server/`,
потому что требует `lowdb`/`lowdb/adapters/FileSync` тем же способом, что
и `main.js` — избегаем повторной настройки пакета и гарантируем
совместимость форматов.

## ⚠️ readiness-calc.js — известный риск рассинхронизации

`mcp-server/readiness-calc.js` — это **копия**, а не общий модуль:
исходник (`src/modules/readiness-calc.js`) написан для `<script>` в
`index.html` без `module.exports`, поэтому server.js грузит копию через
`vm.runInContext` (см. комментарий в server.js). Настоящего общего
источника правды между приложением и MCP-сервером для этого файла нет —
при правке формул готовности в основном приложении копию нужно обновлять
руками.

**Обнаружено 14.07.2026**: копия в `mcp-server/` успела отстать от
оригинала — в оригинале уже появилась `calcChopReadiness()` (модуль ЧОП),
которой в копии не было. Синхронизировано вручную в этой же сессии.
`get_client_readiness` пока не запрашивает ЧОП-готовность, поэтому
расхождение не проявлялось как баг — но при добавлении такого запроса в
будущем нужно ПЕРВЫМ делом свериться, что копия актуальна, а не
доверять этому файлу молча.

## Три независимых предохранителя записи

Все три должны быть выполнены одновременно, иначе `generate_document_package`
честно откажет с понятным сообщением, а не сделает что-то непредсказуемое:

1. **`settings.mcp_enabled === '1'`** — проверяется дважды: при старте
   сервера (server.js) и заново при каждом вызове (mcp-write-bridge.js) —
   на случай, если флаг выключили, пока сервер уже был запущен.
2. **`settings.backup_path` настроен и существует** — без него генерация
   через MCP отказывает. Бэкап делается автоматически ПЕРЕД записью,
   каждый раз, с уникальным именем (`kompliance_backup_mcp_<timestamp>.json`).
3. **Файловая блокировка** (`lock-utils.js`) — на время всей операции
   другой процесс (например, открытое основное приложение) не может
   писать в ту же базу одновременно. Проверено на реальном
   кросс-процессном тесте (два независимых Node-процесса).

## Что протестировано (на имитации, не на боевой базе)

- `doc-generation.js` изолированно — 3 сценария
- `lock-utils.js` — честный кросс-процессный тест
- Связка `withLock` + `generateDocsForClient` — именно то, что теперь в `main.js`
- `mcp-write-bridge.js` целиком — 3 сценария: флаг выключен (отказ),
  флаг включён но бэкап не настроен (отказ), всё настроено (успешная
  генерация + повторный вызов корректно показывает `unchanged`)

**Не тестировалось** — реальная связка с настоящим `lowdb` (в песочнице
нет доступа к npm-registry, использовалась честная функциональная
имитация той же chain-API). Первый реальный прогон на твоей машине —
это первая по-настоящему сквозная проверка с настоящим пакетом.

## Установка / первый запуск

```bash
cd mcp-server
npm install
```

Перед использованием `generate_document_package`:
1. Открой приложение → настрой папку для резервных копий (то, что уже
   спрашивает `backup:now`, если ещё не настроено)
2. Включи `settings.mcp_enabled` (`'1'`)

## Инструменты

| Инструмент | Read/Write | Что делает |
|---|---|---|
| `list_clients` | Read | Список клиентов с фильтрами |
| `get_client_readiness` | Read | % готовности по модулям |
| `get_overdue_documents` | Read | Просроченные/приближающиеся события |
| `generate_document_package` | **Write** | Формирует пакет документов клиента — работает и при закрытом приложении |

## Архитектурные ограничения (не нарушать)

1. `generate_document_package` не сочиняет текст документа — вызывает
   тот же `generateDocsForClient`, что и кнопка в интерфейсе.
2. Любая запись в базу — только через `withLock`. Никогда не пишем в
   `kompliance.json` напрямую, минуя блокировку.
3. Без настроенного бэкапа — запись запрещена, без исключений.
4. Ответы read-инструментов по умолчанию агрегированы (числа, статусы),
   не содержат персональных данных физлиц.
